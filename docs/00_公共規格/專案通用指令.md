# 專案通用開發指令與守則 (給 AI Agent / Antigravity)

本文件定義了 AI 在協助開發 `insurance_helper` 專案時，必須遵循的通用開發守則、技術規範與 UI/UX 設計標準。

---

## 🎨 UI/UX 與視覺設計守則

1. **設計美學優先 (Premium Aesthetics)**:
   - UI 介面必須看起來極具質感、現代且精緻。避免使用瀏覽器或作業系統的陽春預設風格。
   - 使用精心調配的和諧調色盤（例如：漸層、深色模式、毛玻璃效果/Glassmorphism），而非單調的純紅、純藍或純綠。
   - 字體應選用現代且清晰的字型（例如 Google Fonts 的 Inter, Outfit 或 Roboto）。

2. **跨平台技術規範**:
   - **網頁/Web 元件開發**: 
     - 除非使用者另有要求，否則請優先使用 **Tailwind CSS** 作為樣式框架，以利快速且具彈性地調整版面。
     - 遵循語意化 HTML5 標籤，確保良好的可讀性與 SEO。
   - **Flutter (Dart) 行動端開發**:
     - 必須完全啟用 `uses-material-design: true` 並遵循 **Material Design 3 (M3)** 設計規範。
     - 保持元件高度模組化、響應式（Responsive Layout），使其能在不同尺寸的 iOS / Android 手機及平板上完美呈現。
     - 在 UI 元件中加入細微的微動畫 (Micro-animations) 與懸停/點擊反饋 (Hover/Tap effects)，提升操作趣味與流暢度。

---

## 💻 程式碼品質與架構規範

1. **狀態管理 (State Management)**:
   - 在專案擴展時，應優先與使用者確認所要採用的狀態管理套件（如 Riverpod, Provider 或 Bloc）。
   - 避免直接將業務邏輯寫在 UI Widget 中，落實介面（View）與邏輯（Logic/Controller）分離。

2. **Supabase 整合規範**:
   - 所有的資料庫存取皆應遵循 Row Level Security (RLS) 的設計限制。
   - 存取 Supabase 時應封裝成 Repository 或 Service 類別，避免在 UI 直接調用 `Supabase.instance.client`。

3. **保留既有代碼與註釋 (Code Integrity)**:
   - 修改現有代碼時，**必須保留**無關變更的現有註釋、文件說明與程式邏輯，除非使用者明確要求刪除。
   - 新增程式碼時，請撰寫清晰易懂的繁體中文 Dartdoc/註釋，特別是複雜的商業邏輯與 AI 分析區塊。

---

## 🧪 驗證與測試流程

- 每次完成功能開發後，AI 必須執行或提示執行對應的測試指令：
  - Flutter 單元測試 / 整合測試：`flutter test`
  - 確保專案的靜態分析無錯誤：`flutter analyze`
- 在提交程式碼前，確保程式碼通過 linter 檢查（遵循 `analysis_options.yaml` 的規範）。

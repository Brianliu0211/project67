# Git 提交訊息規範 (Commit Message Convention)

本文件定義專案人員在 GitHub Desktop（或其他 Git 工具）提交程式碼時，Summary 與 Description 的標準格式。
目的是讓每一次的 Git 提交紀錄都**簡潔、一致、看得懂**，不會因為表達方式不同而造成日後回頭查閱時的困惑。

---

## 📝 Summary 格式（必填）

```
<類型>: <簡短白話說明>
```

### 可選的類型列表

直接從下表挑選一個最符合的類型，貼在 Summary 的開頭：

| 類型 | 什麼時候用 | 範例 |
|---|---|---|
| `feat` | 新增了一個功能或頁面 | `feat: 新增客戶卡片清單頁面` |
| `fix` | 修復了一個 Bug | `fix: 修復登入頁按鈕點不到的問題` |
| `docs` | 只改了文件（沒動到程式碼） | `docs: 更新工具包版本資訊` |
| `style` | 只改了外觀或排版（沒改功能邏輯） | `style: 調整客戶卡片的字體大小與間距` |
| `env` | 環境設定、套件更新、系統設定 | `env: 升級 supabase_flutter 至 v2.17.0` |
| `refactor` | 重構程式碼（功能沒變但寫法改了） | `refactor: 將資料庫存取抽成 Repository 類別` |
| `test` | 新增或修改測試程式碼 | `test: 新增客戶 CRUD 的單元測試` |
| `chore` | 雜項（清理檔案、調整設定等） | `chore: 清除暫存測試檔案` |

### 混合類型

如果一次提交包含多種類型的變更，用 `&` 或 `+` 連接：
```
docs & env: 蘿蔔桌機環境建置完成與工具包更新
feat + style: 新增客戶清單頁面與卡片視覺設計
```

---

## 📋 Description 格式（選填但建議填寫）

用**條列式白話文**說明「這次改了什麼」，每一項用數字編號：

```
1. 新增客戶清單頁面，使用 ListView.builder 渲染客戶卡片。
2. 串接 Supabase 的 customers 資料表，實作即時資料載入。
3. 新增下拉刷新功能 (RefreshIndicator)。
```

### 撰寫原則
- **白話優先**：不用寫得很專業，用「跟夥伴解釋你剛才做了什麼」的語氣就好。
- **具體但簡短**：每一項控制在一行以內。
- **不需要解釋為什麼**：Description 只需要說「做了什麼」，原因留給開發日誌。

---

## 🤖 AI 自動產出

當您對 AI 說「**收工**」時，AI 會自動幫您產出符合本規範的 Summary 與 Description，您只需要直接複製貼上到 GitHub Desktop 即可。

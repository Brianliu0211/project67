# N8N 專題整合與架構評估報告

**建立日期**：2026-07-21  
**評估標的**：`insurance_helper` 專案導入 N8N 工作流自動化引擎之可行性、利弊與潛在隱患

---

## 📌 1. 專題背景與問題說明

原專題組員先前已有開發 N8N 相關內容之經驗。當初在舊專題使用 N8N 時，遭遇的核心痛點為 **「LLM Token 數嚴重不足與耗盡」** 以及 **「將整個前端請求過載於 N8N 導致不穩定性過高」**。

本報告旨在評估：
1. **完全不使用 N8N（純 Flutter + Supabase）** vs **混合導入 N8N 自動化** 兩方案之詳細優缺點與隱患。
2. 是否有將 N8N 整合進目前 `insurance_helper` 專案之必要性？
3. 如何為該組員規劃簡單、獨立且高價值的角色定位與任務？
4. 對專案未來規劃之衝突（風險）與幫助評估。

---

## ⚖️ 2. 兩大架構方案深度對比

### 方案一：完全不使用 N8N（純 Flutter + Supabase 原生開發）

#### ⚙️ 運作方式
所有功能（客戶資料管理、Google 日曆同步、LINE/Email 提醒、語音轉錄與 Gemini 語意分析）完全由團隊在 **Flutter 前端**、**Dart Packages** 與 **Supabase (PostgreSQL / Edge Functions / pg_cron)** 中手寫程式碼串接 API 實作。

#### 👍 優點 (Pros)
1. **架構極致純粹與穩定**：系統組件少（僅 App + Supabase），無第三方中繼站，網路延遲最低，不會因中間伺服器停擺而影響主功能。
2. **完全掌控與標準版本控制**：所有邏輯都在 Git 程式碼倉庫中，修改記錄透明，支援嚴格的 Code Review 與 CI/CD 自動化測試。
3. **零額外託管費用**：無需營運、維護或購買 N8N 伺服器 (Docker/Cloud Server)，無第三方平台免費額度過期壓力。
4. **資安風險較集中**：所有 API Key 與 Access Token 均存放在 Supabase 安全環境或 `.env` 中，無中繼平台洩漏金鑰之疑慮。

#### 👎 缺點 (Cons)
1. **開發工作量大幅暴增**：
   * 串接 Google 日曆手寫 OAuth2 授權、Token 自動刷新 (Refresh Token) 及 API HTTP 請求代碼耗時耗力。
   * 發送 LINE / Email 提醒需手寫 Supabase Edge Functions (Deno/TypeScript) 或 Dart 串接套件。
2. **組員分工困難**：若該組員不熟悉 Flutter/Dart 或 Supabase 後端開發，將難以實質參與核心程式碼寫入。

#### ⚠️ 隱患 (Hidden Risks)
1. **Google OAuth 審查與動態權限地獄 (Dynamic Scope & Restricted Scope Hell)**：在 Flutter App 內直接串接 Google Calendar / Drive，極易觸發 Google 嚴格的敏感權限審查，導致用戶登入時出現「未經驗證應用程式」警告。
2. **手機/瀏覽器背景任務失效風險**：提醒功能若完全依賴 Flutter App 本地端，一旦手機關閉 App、啟用省電模式或 Web 瀏覽器分頁關閉，定時推播提醒可能直接失靈。
3. **API 變更的維護負擔**：第三方 API 規格變更時，必須重新編譯發布 Flutter App 或部署 Edge Functions。

---

### 方案二：導入 N8N（Flutter/Supabase 核心 + N8N 後台自動化）

#### ⚙️ 運作方式
* **核心功能**（App UI、客戶 CRUD、Gemini 語音轉錄分析、權限驗證）依然由 Flutter + Supabase 負責。
* **周邊自動化與串接**（發送 LINE 提醒、同步至 Google 日曆、自動生成每週 PDF 週報）由 Supabase Webhook 或 N8N 內部定時器發送給 **N8N**，由 N8N 的視覺化工作流完成執行。

#### 👍 優點 (Pros)
1. **開發速度極快 (Low-Code)**：串接 Google 日曆、LINE、Gmail 等，N8N 提供現成 Node，拖拉流程 10~30 分鐘即可完成，完全免寫 OAuth 授權與 API 套件代碼。
2. **完美的組員獨立分工**：該組員可 **100% 獨立負責 N8N 流程開發與測試**，無需碰撞複雜的 Flutter/Dart 代碼，大幅降低溝通與技術門檻。
3. **擴充性極佳**：未來若增加「發送 Telegram 訊息」或「同步 Notion 備忘錄」，僅需在 N8N 增加節點，前端程式碼完全無須動彈。

#### 👎 缺點 (Cons)
1. **多了一層系統依賴 (External Dependency)**：若 N8N 伺服器網路中斷、記憶體溢出或帳號過期，相關自動化提醒/同步功能會瞬間失效。
2. **須維護 N8N 託管環境**：N8N Cloud 有免費試用與額度限制；若自行用 Docker 託管在 Render / Railway / AWS，需花費微量維護心力與成本。
3. **追查問題鏈結變長 (Debugging Difficulty)**：若提醒未發出，排查路徑增加（Flutter ➔ Supabase Webhook ➔ N8N 伺服器 ➔ 外部 API）。

#### ⚠️ 隱患 (Hidden Risks)
1. **Token 暴增與 API Rate Limit 刷爆陷阱（舊專題痛點）**：
   * **隱患細節**：若組員流程設計不當（如無限迴圈 Loop、Cron 觸發頻率過高），或**重蹈覆徹在 N8N 節點中呼叫 LLM 處理大篇幅全文**，可能於短時間內刷爆 Gemini API Token 額度。
   * **防範解法**：**嚴禁在 N8N 中做 AI 大文本分析**。N8N 僅接收 Supabase 提煉完成之精簡 JSON，做條件分流與邏輯判斷。
2. **Webhook 暴露與資安洩漏風險**：N8N 開放的 Webhook URL 若無設定 `Header Secret Auth Token` 安全機制，外部駭客可能掃描發起惡意請求。
3. **金鑰託管風險**：N8N 伺服器內儲存之 Google/LINE/Supabase 存取 Key，若存取密碼過於簡單可能成為資安破口。

## 📊 3. 兩方案綜合評估對比表

| 評估項目 | 方案一：全自研 (無 N8N) | 方案二：混合模式 (使用 N8N) | 勝出方案與理由 |
| :--- | :--- | :--- | :--- |
| **開發效率與速度** | 🐢 較慢（手寫 OAuth / 複雜 API） | ⚡ 極快（拖拉式視覺化 Node 串接） | **方案二 勝** |
| **組員分工與產出** | 🔴 困難（不熟悉 Flutter 者難有產出） | 🟢 最佳（組員可 100% 獨立負責 N8N） | **方案二 勝** |
| **系統架構簡潔度** | 🟢 高（僅 App + Supabase） | 🟡 中（App + Supabase + N8N 三方） | **方案一 勝** |
| **Token 耗盡風險** | 🟢 極低（代碼精準控制請求） | 🔴 中高（若 N8N 流程設計不當易爆量） | **方案一 勝 (方案二需設防呆門檻)** |
| **背景提醒可靠度** | 🔴 較受限（受限 App/瀏覽器存活） | 🟢 較高（伺服器等級 Cron 獨立運作） | **方案二 勝** |
| **營運成本 (費用)** | 🟢 零額外費用 | 🟡 需微量伺服器成本或免費額度管控 | **方案一 勝** |

## 👤 4. 該組員的角色定位與「簡單高價值」任務規劃

建議讓該組員負責 **「後台非同步工作流與外部串接引擎」**，推薦 3 個簡單、獨立且高價值的實作方向：

1. **📅 自動同步行程至 Google Calendar / Outlook**
   * **流程**：Supabase Webhook ➔ N8N Google Calendar Node ➔ 自動建立日曆行程。
   * **價值**：免去 Flutter 端編寫 Google OAuth2 與 API 的複雜度。
2. **🔔 多管道跟進提醒推播 (LINE / Email / Telegram)**
   * **流程**：N8N 定時器 (Cron) ➔ 撈取 Supabase 待提醒資料 ➔ LINE Notify / Gmail 自動推播給業務員。
   * **價值**：解決行動裝置背景通知失效問題，提升提醒到達率。
3. **📈 業務員每週/每月自動化摘要週報發送**
   * **流程**：每週日 Cron 觸發 ➔ 彙整本週新增客戶與紀錄 ➔ 產出 Email/PDF 週報發送給業務員備份。

## 💡 5. 總結與決策方針

1. **若選擇「方案一 (完全不用 N8N)」**：團隊需自行承擔 LINE/Google Calendar 的所有 Flutter 代碼編寫，且該組員較難有獨立程式碼產出。
2. **若選擇「方案二 (適度導入 N8N)」**：此為 **「大幅減少開發工作量 + 讓該組員擁有高價值獨立貢獻」** 的優質解答。
3. **安全落地兩大鐵律**：
   * 🛑 **鐵律一 (解決 Token 暴增)**：N8N **絕不處理**原始語音或大篇幅文字 AI 提煉，只接收 Supabase 處理好的 JSON 做條件分流。
   * 🛑 **鐵律二 (資安與防護)**：Supabase 呼叫 N8N Webhook 必須附帶 Header Secret 驗證，且 N8N 故障時不影響 Flutter 前端網頁主流程操作。

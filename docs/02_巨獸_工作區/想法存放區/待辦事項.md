## 待開發功能

### 1. 讓使用者自行上傳照片功能
- **描述**：
  目前系統中的客戶卡片僅支援輸入外部圖片網址 (URL) 作為頭像。為了提升易用性，需支援使用者直接上傳本地照片，並儲存至 Supabase Storage，自動產生可存取的 URL。
- **實作步驟建議**：
  1. **Supabase Bucket 設定**：
     - 在 Supabase Console 建立一個名為 `customer-photos` (或 `avatars`) 的 Storage Bucket。
     - 設定 RLS 政策，允許已驗證的使用者 (authenticated users) 進行上傳 (`INSERT`)、修改 (`UPDATE`) 及刪除 (`DELETE`)，並允許公開讀取 (`SELECT`)。
  2. **套件引入**：
     - 於 `pubspec.yaml` 引入 `image_picker` (或 `file_picker`) 套件以支援在 Web / App 上選取圖片。
     - *注意：若引入新套件，須依規範同步更新 `docs/工具包.md`。*
  3. **前端 UI 與邏輯實作**：
     - 在 `customer_management_tab.dart` 的新增/編輯客戶對話框中，將「照片網址」欄位優化或替換為「選擇並上傳照片」按鈕。
     - 當使用者選取照片後，顯示本地預覽。
     - 點擊儲存時，先將照片檔案上傳至 Supabase Storage，取得 Storage Public URL，再將該 URL 寫入資料庫 `customers.avatar_url` 欄位。
     - 處理上傳時的 Loading 狀態與錯誤異常提示 (以 Toast 通知)。
  4. **功能驗證**：
     - 測試多種常見圖片格式 (.jpg, .png)。
     - 驗證 Web 端上傳與 RLS 權限是否運作正常。

### 3. 切換畫面或修改資料時整個頁面重整與重複讀取資料庫問題
- **描述**：
  目前使用者在系統中切換選單（例如從「今日行程」切換至「客戶管理」），或是新增/修改/刪除客戶資料後，整個客戶管理分頁都會重置，並重新向 Supabase 資料庫發送請求讀取全部資料。這會造成頁面閃爍、讀取延遲以及不必要的 API 消耗。
- **可能原因分析**：
  1. **切換選單重建 Widget**：在 [home_screen.dart](file:///c:/GitHub/project67/lib/screens/home_screen.dart#L286-L292) 中，主工作區域直接使用三元運算子判定是否渲染 `CustomerManagementTab`。切換選單會使該 Widget 被銷毀並重新創建，從而每次都重新執行其 `initState` 裡的 `_fetchCustomers()`。
  2. **修改資料重新 Fetch**：在 [customer_management_tab.dart](file:///c:/GitHub/project67/lib/screens/customer_management_tab.dart) 中，新增、修改、刪除操作成功後，都會呼叫 `await _fetchCustomers();` 重新從資料庫獲取整份名單，而非在本地 State 直接更新對應的項目。
- **待辦細項**：
  - [ ] **狀態保存與 Widget 快取**：
    - 評估在 [home_screen.dart](file:///c:/GitHub/project67/lib/screens/home_screen.dart) 使用 `IndexedStack`、`AutomaticKeepAliveClientMixin` 或將客戶資料狀態提升（State Hoisting）至父組件，以避免切換選單時銷毀 `CustomerManagementTab` 並重複發送 API 請求。
  - [ ] **本地狀態同步更新（Optimistic Updates / Local State Sync）**：
    - 改寫新增/修改/刪除邏輯，在 API 呼叫成功後，直接在本地的 `_allCustomers` 列表中新增、更新或移除該筆資料，並觸發 `setState()` 更新 UI，取代每次都呼叫 `_fetchCustomers()` 重新下載整張表。

### 4. 支援 Google 帳號第三方登入與連結
- **描述**：
  提供使用者在註冊與登入時，能夠直接連結或透過 Google 帳號進行第三方登入（OAuth），以提升登入便利性並簡化註冊流程。
- **實作步驟建議**：
  1. **Google Cloud Console 設定**：
     - 前往 Google Cloud Console 建立 OAuth 2.0 用戶端憑證（需包含 Web、Android、iOS 等平台憑證，依專案支援平台而定）。
     - 設定授權的重新導向 URI（Redirect URI），指向 Supabase 的 Auth 回呼地址。
  2. **Supabase Console 設定**：
     - 在 Supabase Dashboard -> Authentication -> Providers 啟用 Google 登入。
     - 填入 Google OAuth 憑證（Client ID 與 Client Secret）。
  3. **Flutter 程式碼串接**：
     - 在 [login_screen.dart](file:///c:/GitHub/project67/lib/screens/login_screen.dart) 增加「使用 Google 帳號登入/註冊」的按鈕。
     - 呼叫 `supabase.auth.signInWithOAuth` 進行 Google 驗證：
       ```dart
       await supabase.auth.signInWithOAuth(
         OAuthProvider.google,
         redirectTo: kIsWeb ? null : 'my.scheme://login-callback', // 需依平台設定
       );
       ```
     - 處理網頁端與行動端 OAuth 導回（Callback）的會話接收與狀態更新。
- **待辦細項**：
  - [ ] 於 Google Cloud 及 Supabase 設定 OAuth 憑證與 Redirect URIs。
  - [ ] 在 `login_screen.dart` 中刻劃 Google 登入按鈕 UI（遵循 Google 品牌標誌規範）。
  - [ ] 實作 `signInWithOAuth` 邏輯，並確保登入成功後正確同步使用者 Profile 資料。

### 10. 電子名片生成與 Google 雲端硬碟匯出功能
- **描述**：
  在「設定 > 帳號」底下新增「電子名片」功能。系統將根據使用者勾選的基本資料（如：姓名、綽號、電話、信箱、頭像等），動態生成可供匯出的電子名片。此功能亦需支援將產出的名片檔案直接上傳並儲存至使用者的 Google 雲端硬碟（Google Drive）中。
- **作法與設計規劃**：
  本功能將從兩個作法進行研究與參考：
  1. **作法一：動態圖形名片（PNG/PDF 導出）**：
     - **參考相似設計**：在 Flutter 介面上渲染一張精美的卡片，使用者勾選要顯示的欄位後，卡片會動態更新預覽（類似目前客戶卡片的翻轉與精緻排版設計）。
     - 使用 `screenshot` 套件將該 Widget 轉為圖片，或使用 `pdf` 套件將卡片繪製為 PDF 格式。
  2. **作法二：標準通訊錄名片（vCard / .vcf 檔案導出）**：
     - 將勾選的資訊格式化為標準的 vCard 3.0 或 4.0 規格文字，儲存為 `.vcf` 檔案，讓收件人點擊後能直接將聯絡人匯入至手機或 Google Contacts。
  3. **Google 雲端硬碟 (Google Drive) 連接**：
     - 整合 `google_sign_in` 與 `googleapis` 兩個 Dart/Flutter 套件。
     - 使用者授權 Google Drive 存取範圍（Scope: `https://www.googleapis.com/auth/drive.file`）。
     - 在應用程式內呼叫 Google Drive API，在使用者雲端硬碟中自動建立專屬資料夾並將生成的電子名片檔案（.png/.pdf/.vcf）上傳儲存。
- **待辦細項**：
  - [ ] 於 `pubspec.yaml` 引入 `google_sign_in`、`googleapis` 以及相關導出套件（如 `screenshot` 或 `vcard`）。
  - [ ] 在「設定 > 帳號」下設計勾選表單與動態卡片預覽 UI。
  - [ ] 實作動態名片檔產生邏輯（vCard 格式或 PNG/PDF 圖檔）。
  - [ ] 實作 Google 帳號授權與 Google Drive API 檔案上傳對接。

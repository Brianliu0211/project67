# 客戶進度追蹤功能 (Customer Progress Tracking) 設計方案

本文件詳細規劃了 **「客戶進度追蹤」** 功能的三種實作方案。此功能旨在協助保險業務代表更清晰地管理每位客戶目前的銷售階段（Pipeline）、掌握核保與跟進流程，並預估成交時間。

---

## 📌 方案一：客戶詳情時間軸與檢查清單 (Customer Timeline & Stepper)

此方案專注於**單一客戶的微觀進度管理**，直接整合在現有的客戶卡片放大詳情對話框中。

### 🎨 開發完成後長怎樣？
1.  **卡片正面視覺**：客戶卡片正面會新增一個精簡的小型進度指示標籤（例如：`建議書規劃中` 或 `核保照會中`），使用帶有透明度的品牌色調呈現。
2.  **詳情對話框 (Zoom Detail Dialog)**：
    *   點開詳細資訊後，上方會出現一條橫向的**步驟進度條 (Stepper)**：
        $$\text{初次接觸} \rightarrow \text{需求分析} \rightarrow \text{建議書規劃} \rightarrow \text{建議書解說} \rightarrow \text{核保送件} \rightarrow \text{保單簽收 (結案)}$$
    *   目前所處的階段會以亮綠色/科技藍高亮顯示，已完成的階段顯示打勾圖示，未開始的階段呈半透明灰色。
    *   每個階段下方都有對應的 **SOP 檢查清單 (Checkbox)**。例如當進度在「核保送件」時，下方會顯示：
        *   `[x] 填寫要保書`
        *   `[x] 首次保費扣款設定`
        *   `[ ] 客戶體檢確認`
        *   `[ ] 照會補件 (若有)`
3.  **時間與天數預估**：進度條旁會顯示 `此階段已停留 5 天`，以及根據設定顯示 `預估成交日：2026/08/10`。

### 🖱️ 如何使用？
1.  **切換階段**：業務點擊進度條上的任何節點，系統會跳出確認視窗：「是否將此客戶移至 [建議書解說] 階段？」，確認後卡片狀態即時更新。
2.  **勾選任務**：業務在目前階段下，每完成一項 SOP 工作（例如體檢完畢），直接在對話框內勾選對應的 Checkbox，進度條下方的進度比例（例如 `2/4 完成`）會動態跑條。
3.  **自動提示**：當目前階段的所有 Checkbox 都被勾選完畢時，進度條會閃爍微光，提示業務「所有準備工作已完成，建議進入下一階段」。

## 📌 方案二：視覺化銷售漏斗看板 (Sales Pipeline Kanban Board)

此方案專注於**全盤業務的巨觀管理**，在系統中新增一個獨立的分頁，以看板（Kanban）形式呈現。

### 🎨 開發完成後長怎樣？
1.  **獨立分頁**：側邊欄新增一個「銷售漏斗」分頁，點擊後會進入一個橫向捲動的看板畫面（類似 Trello 的排列方式）。
2.  **看板欄位 (Columns)**：看板由左至右分別為不同的銷售階段欄位（從「初次接觸」到「保單簽收」）。
3.  **客戶卡片 (Cards)**：
    *   每個欄位下方整齊排列著屬於該階段的客戶簡短卡片（包含客戶姓名、大頭貼、標籤以及一小條進度比例條）。
    *   每個欄位上方會顯示該欄位目前的客戶總數，以及預估的潛在業績總額（例如：`核保送件中 (5人) | 預估保費 NT$150,000`）。

### 🖱️ 如何使用？
1.  **拖拽變更進度**：業務代表可以直接用滑鼠（或在平板上用手指）點按某張客戶卡片，將其拖曳（Drag and Drop）到隔壁的欄位，即可瞬間變更該客戶的銷售階段。
2.  **快速篩選與搜尋**：看板上方提供快速篩選功能，例如「只看高意願客戶」或「只看車險客戶」，看板會以平滑動畫重新排列過濾後的卡片。
3.  **點擊卡片開啟詳情**：在看板上直接點擊某個客戶卡片，依然會彈出該客戶的「詳情對話框」，以便業務查看詳細備註與填寫提醒。

## 📌 方案三：AI 智能語意自動推進 (Gemini-Powered Smart Progress)

此方案是結合 **Phase 4 (Gemini API 語意分析)** 的智慧型功能，免去業務代表手動手動調整進度的繁瑣。

### 🎨 開發完成後長怎樣？
1.  **AI 進度助理浮窗**：在首頁或客戶管理頁面，系統會新增一個「AI 進度小幫手」的動態提示區塊。
2.  **智慧變更通知卡片**：當系統透過語音備忘錄解析出客戶的進度變化時，畫面上會彈出一個半透明的精美對話框：
    > **✨ AI 進度建議**
    > 偵測到您在對 **林小明的語音備忘** 中提到：*「今天跟林小明談完了建議書，他決定要規劃這套方案，約了明天去填要保書。」*
    > 
    > 💡 **建議操作**：
    > *   將 **林小明** 的進度從 `建議書規劃` $\rightarrow$ 推移至 `核保送件`。
    > *   自動建立明日日程：`填寫要保書`。
    > 
    > [ 忽略 ]  [ 一鍵套用 ]

### 🖱️ 如何使用？
1.  **正常輸入備註**：業務代表像平常一樣，在語音錄音後讓系統轉錄為文字，或者手動在卡片背面（[客戶卡片備註程式碼](file:///c:/GitHub/project67/lib/screens/customer_management_tab.dart#L1240-L1250)）輸入拜訪紀錄。
2.  **後端自動分析**：存檔時，Supabase Database Webhook 會將更新的文字發送給 Gemini API。Gemini 分析文字後，若判定有符合階段移轉的關鍵描述（如「同意簽約」、「約時間填要保書」、「收到照會要求」等），便會生成建議。
3.  **一鍵確認推進**：業務在 App 首頁或客戶卡片上看到 AI 建議時，只需點擊 **「一鍵套用」**，系統便會自動在背景修改資料庫、移動看板位置、並同步建立行事曆日程，省去手動新增與拖拽的時間。

## 🗄️ 資料庫與架構預估 (Database & Architecture)

為了在未來支援這三種方案，我們需要在資料庫結構做以下輕量化設計：

1.  **`customers` 資料表變更**：
    *   新增 `current_stage` (VARCHAR 或 ENUM)：代表目前銷售階段。
    *   新增 `stage_updated_at` (TIMESTAMPTZ)：用於計算在此階段停留了多久。
2.  **新增 `customer_checklists` 資料表（非必選，可用 JSONB 代替）**：
    *   紀錄每個客戶在各階段的 SOP 完成狀況。為求精簡，亦可直接以一個 `JSONB` 欄位儲存於 `customers` 表中，如：
        ```json
        {
          "underwriting": {
            "apply_form": true,
            "medical_check": false
          }
        }
        ```
3.  **與垃圾桶機制的相容性**：
    *   這些進度追蹤表與欄位，都必須遵循我們之前設計的垃圾桶機制（[待辦事項 - 垃圾桶設計說明](file:///c:/GitHub/project67/docs/02_%E5%B7%A8%E7%8D%B8_%E5%B7%A5%E4%BD%9C%E5%8D%80/%E6%83%B3%E6%B3%95%E5%AD%98%E6%94%BE%E5%8D%80/%E5%BE%85%E8%BE%A6%E4%BA%8B%E9%A0%85.md#L204-L227)）。只有 `deleted_at IS NULL` 的活動客戶才會顯示在進度看板中。

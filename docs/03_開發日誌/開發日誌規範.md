# 🚀 專案開發日誌撰寫規範 (AI Assistant Developer Log Specification)

## 📌 核心宗旨
本規範旨在約束 AI 助理（Antigravity）在完成任何階段性程式碼變更、功能實作或系統架構調整後，根據使用者指令在 `docs/03_開發日誌/` 目錄下建立獨立開發日誌的行為。
日誌必須具備「極高的技術透明度」與「高可讀性」，同時滿足工程師（技術驗證）與 PM（進度追蹤）的需求。

---

## 🛠️ 撰寫鐵律 (Core Rules)

1. **實證與代碼優先 (Code-First & Terse)**：
   - 嚴禁空泛的高層次文字描述（例如：優化了介面、提升了安全性）。
   - 必須精確指出「修改了哪些檔案」、「新增了哪些元件」、「調整了哪些關鍵邏輯/API」，並附上幾行關鍵代碼或結構變更明細。
2. **規格書與模組對齊 (Feature Alignment)**：
   - 所有功能的更新，必須嚴格對照 `docs/00_公共規格/main_spec.md` 內定義的模組名稱與編號（如：【模組一：基礎建設與安全防護】、【模組五：業務戰情中樞與排程】）。
3. **明確區分「主線」與「沙盒」 (Sandbox Isolation)**：
   - AI 自行記錄時，必須明確指出本次變更是直接作用於主專案，還是處於 `docs/01_蘿蔔_工作區/AI_對話實驗室/` 或 `docs/02_巨獸_工作區/AI_對話實驗室/` 的沙盒測試階段。
4. **環境與套件異動追蹤 (Environment Tracking)**：
   - 若變更涉及 `pubspec.yaml`、環境變數（`.env`） or 資料庫結構（`*.sql`），必須在日誌中以【⚠️ 破壞性變更/環境異動】粗體高亮標出。

---

## 📝 開發日誌標準格式 (Markdown Template)

每次新建開發日誌檔案時，必須採用以下結構填寫：

```markdown
# 📅 [YYYY-MM-DD] - 【模組名稱與編號】功能標題

> **執行狀態**：🟢 已併入主線 (src) / 🟡 沙盒實驗階段 (docs/沙盒路徑)

## 🎯 實作成果摘要
- 一句話說明本次變更為用戶或系統帶來什麼具體改變。

## 💻 技術變更明細 (Actual Technical Changes)
- **[NEW / MODIFY] `路徑/至/檔案.dart`**：
  - 具體描述修改了什麼邏輯或元件名稱。
  - 範例代碼塊（若為關鍵重構或 API 連接必填）：
    ```dart
    // 秀出最核心的 3-5 行變更程式碼
    ```
- **[DB] `supabase_schema.sql`**：
  - 記錄資料庫欄位或 RLS 政策的增刪。

## ⚠️ 環境異動與破壞性變更 (Environment & Breaking Changes)
- 若無，請填「無」。
- **套件異動**：新增/刪除/升級了哪些套件（如 `flutter_dotenv: ^6.0.1`）。
- **密鑰異動**：`.env` 是否新增了預留變數。

## 🚦 團隊協作與驗證指南 (Verification Plan)
- **工程師驗飾**：提供本地測試的具體指令（如 `flutter test` 或本機伺服器指令）。
- **PM/夥伴預覽**：說明拉取（Pull）程式碼後，如何在不安裝複雜環境下直接看到或測試該功能（例如提供 Mock Data 測試按鈕說明）。
```
